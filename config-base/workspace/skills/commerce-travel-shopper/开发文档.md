# Commerce Travel Shopper 开发文档

## 1. 目标与范围

本文档用于指导 `commerce-travel-shopper` 的工程实现，目标是交付一个“可代理执行、关键节点人工确认”的比价与下单助手。

平台范围：
- 机票/酒店：飞猪、去哪儿、携程、美团
- 商品：淘宝、京东

数据方式：无 API 前提下，以截图采集 + OCR 结构化为主。

交互渠道：**飞书聊天会话（私聊/群聊）**，默认采用“交互卡片优先、文本命令兜底”。

---

## 2. 架构总览

建议采用 6 个模块：

1. **Task Planner**
   - 接收任务并生成平台级采集计划
2. **Collector Agent**
   - 浏览器执行与截图采集
3. **OCR Parser**
   - 抽取价格、税费、退改/取消规则、评分等
4. **Normalizer**
   - 统一字段模型，便于跨平台比较
5. **Ranking Engine**
   - 按场景（机票/酒店/商品）评分排序
6. **Execution Guard**
   - 控制状态机与人工确认节点

---

## 3. 状态机设计

```text
COLLECTING
  -> ANALYZING
  -> RANKED
  -> DRAFT_READY
  -> WAIT_CONFIRM_SUBMIT
  -> WAIT_CONFIRM_PAY
  -> DONE

任意状态 --(ABORT/风控触发)--> ABORTED
```

状态流转约束：
- 未进入 `WAIT_CONFIRM_SUBMIT` 禁止触发“提交订单”操作
- 未进入 `WAIT_CONFIRM_PAY` 禁止触发“支付”操作

会话流转约束（飞书）：
- 机器人已固定绑定目标群，默认以群内上下文执行，不强制额外会话ID映射
- 同一群同一时刻仅允许一个 active task（可配置）
- 群聊场景下，确认动作需校验操作者权限（白名单/发起人/管理员）

建议首版接口：

- `prepare_compare_task(query)`

- `ocr_extract(task_id)`

- `render_pay_confirm_card(task_id)`：渲染支付前确认卡
- `handle_feishu_action(action_payload)`：处理按钮点击回调

## 6. 风控与止损策略

触发即中止（ABORTED）：
1. 解析价格与页面展示偏差 > 5%
2. 退改/取消条款缺失
3. 风控验证码 2 分钟未人工处理
4. 订单总价超预算阈值
5. 用户执行 `ABORT`

回滚动作：
- 停止点击
- 关闭订单页
- 输出中止原因与最后证据截图

飞书侧风控补充：
- 确认按钮默认 10 分钟有效，超时后卡片失效并要求重新确认
- 对重复点击同一确认按钮执行幂等处理（只生效一次）
- 网络抖动重试不重复提交订单（依赖幂等键）

## 7. 平台执行规范

### 7.1 机票（飞猪/去哪儿/携程/美团）
- 采集：列表首屏、价格排序首屏、候选详情
- 关键核验：税费、行李、中转、退改

### 7.2 酒店（飞猪/去哪儿/携程/美团）
- 采集：列表首屏、价格排序首屏、房型详情
- 关键核验：取消政策、是否含早、总价构成

### 7.3 商品（淘宝/京东）
- 采集：列表首屏、详情页、结算价截图
- 关键核验：规格一致性、券后价、运费、售后

## 8. 日志与审计

每个 task 需输出：
- `events.log`：状态变化、确认动作
- `offers.json`：结构化报价
- `report.md`：比价结果
- `evidence/`：截图与索引

建议日志字段：
- `ts`, `task_id`, `state_from`, `state_to`, `action`, `operator`, `result`

飞书审计字段建议补充：
- `message_id`, `card_action_id`（`chat_id/thread_id`可选）
- `operator_open_id`, `operator_name`
- `idempotency_key`, `confirm_token`

## 9. 飞书交互协议（重点）

### 9.1 用户输入协议

两种输入方式并行：

1. **卡片按钮（默认）**
   - `开始比价`、`确认提交`、`确认支付`、`中止任务`
2. **文本命令（兜底）**
   - `SHOP OCR_EXTRACT [task_id]`
   - `SHOP RISK_CHECK [task_id]`
   - `CONFIRM_SUBMIT <task_id>`
   - `CONFIRM_PAY <task_id>`
   - `ABORT <task_id>`
   - 中文别名：`确认提交` / `确认支付` / `取消任务`

### 9.2 消息输出规范

每个阶段输出固定结构：

- 阶段标题（如：`[比价完成]`）
- 当前状态（state）
- 下一步操作（按钮+文本命令）
- 超时提示（例如：`本确认10分钟内有效`）

### 9.3 幂等与防误触

- 为每次“提交/支付确认”生成唯一 `idempotency_key`
- 同一 key 二次请求返回“已处理”而不重复执行
- 如果任务状态已推进，旧卡片按钮点击应返回“卡片已过期，请使用最新卡片”

### 9.4 权限校验

建议策略：
- 私聊：默认发起人可确认
- 群聊：仅发起人 + 授权用户组可确认（例如采购组）
- 支付确认可配置为“二次复核”（两人确认）

## 11. 验收标准

1. 可完成一次跨平台比价并输出报告
2. 提交/支付前无确认不得继续
3. ABORT 能在任意阶段停止执行
4. 每个候选都有证据截图可回溯
5. 飞书卡片确认可用，且重复点击不重复执行
