### 1.3 发现的问题

#### 🔴 P0 — 飞书群新成员@机器人无响应

**根因分析：**

通过审计 `bot.ts` 的消息处理链，发现以下处理流程：

```
消息到达 → dedup检查 → 解析sender → 群组策略检查 → @mention检查 → DM策略检查 → 调度执行
```

**无响应的3个可能原因：**

1. **`groupPolicy: "allowlist"` + `groupAllowFrom` 仅包含群ID，不限制sender** — 群组级别白名单检查的是 `ctx.chatId`（群ID），不是发送者ID。配置中 `groupAllowFrom: ["oc_bb11ddeb722729367b17cbd179a189b4"]` 只允许这一个群。如果新成员在**其他群**@机器人，会被直接 `return` 无任何响应。

2. **`senderNameCache` 权限问题导致静默失败** — `bot.ts:200行` 的 `refreshFeishuSenderName` 需要 `contact:user.base:readonly` 权限。如果飞书应用权限不足，会触发权限错误，虽然不会阻断消息处理，但会在获取sender信息时出现延迟或失败。对新加入的组织成员，如果其 `open_id` 在飞书组织内但应用未获得该用户的权限范围（例如跨部门），可能导致 API 调用异常。

3. **⭐ 最关键原因：`sessionDispatchLanes` 串行化 + 超时死锁** — `bot.ts` 中 `LANE_DISPATCH_TIMEOUT_MS = 90_000`（90秒硬上限）。如果上一条消息的 session 还在处理中（比如Cron任务或其他用户的消息触发了长时间运行），新消息会被放入 lane queue。当 lane 队列积压，且前面的任务超时，后续消息会等待甚至丢失。**特别是当 `sessionDispatchConcurrency: 5` 但实际只有1个session key时，所有群消息共享同一个 dispatch lane**。

4. **`dmPolicy: "allowlist"` 对私信的影响** — 新成员如果先私信机器人，会触发 pairing 流程（发送配对码），但如果用户不理解配对流程，会认为机器人无响应。配置中 `allowFrom` 只有一个 `ou_3665e9a9d79f52105fd5868bae4a8b49`（你本人）。

#### 🟡 P1 — V5认知管道响应时长过长

- **`v5-orchestrator.py --process`** 的 P50 延迟 = **6-7秒**（从 orchestrator-log 可见）
- 即使是简单消息如"早"、"对的"也需要 5-7 秒经过 LLM 分析
- 快速规则(`QUICK_RULES`)仅覆盖 greetings/video/quant/delete_danger/xhs 5种模式，大量日常消息都 fallback 到 LLM
- `thought-quality-gate-v5.py` 仅用 DeepSeek 单 provider，无 fallback

#### 🟡 P1 — 理解能力不足

- `message-analyzer-v5.py` 的 LLM 分析 prompt 固定模板，无上下文感知
- 分析结果未考虑**对话历史**，每条消息独立分析
- 缺少**语义记忆检索**：用户说"上次那个方案"时无法自动关联

#### 🟡 P1 — 推理能力局限

- `multi-step-reasoning-v5.py` 的推理链是**串行单线程**，5步模板固定
- 每步都调完整 LLM，无法利用上一步的 streaming 结果加速
- 推理模板仅5种，无法自适应新场景
- 推理结果无**可验证性评估**（只有粗糙的关键词匹配判断置信度）

#### 🟡 P1 — 自我迭代闭环不完整

- `proactive-iteration-engine-v5.py` 能**发现**问题，但**修复建议**只是文本输出，无自动执行
- `rule-candidates.json` 生成了候选规则，但没有自动注入到 `QUICK_RULES` 或 `pattern-library`
- 错误追踪器有 44 条错误，修复率未统计
- 学习闭环断裂：`.learnings/` → `pattern-library.json` → `QUICK_RULES` 三级提升无自动化

#### 🟡 P1 — 响应时长监控空洞

- `response-latency-monitor.py` 需要**手动调用** `--record` 才能记录延迟
- 无法自动 hook 进飞书消息链路
- 当前只有 7 天内的少量数据点

#### 🟢 P2 — Cron 任务超时

- `V5 API Fallback健康检查`：连续3次超时（60s timeout不够，脚本实际需要3次API调用，每次最长15s）
- `每周综合审计`：超时（240s timeout，脚本可能需要更多时间）
- `综合监控报告`：delivery=`not-delivered`（mode设为none但期望announce）

#### 🟢 P2 — 安全问题

- `openclaw.json` 中 `appSecret` 和 `apiKey` 都是明文存储
- `lossless-claw` 插件配置的 `summaryModel` 引用了 `bailian/qwen3.5-plus`，如果 provider 改名会断裂

## 第二部分：V6 造神计划升级方案

### 2.0 V6 设计理念

**V5 = 认知管道（串行分析）→ V6 = 认知内核（实时融合）**

V5的核心问题是**管道式架构**：每条消息要经过 analyze → plan → quality_gate → finalize 多个独立脚本，每个脚本独立调LLM，导致响应慢、上下文断裂。V6要将这些能力**融合为一个统一的认知内核**，实现：

- 🧠 **上下文感知**：分析时自带对话历史 + 语义记忆
- ⚡ **亚秒决策**：规则引擎覆盖90%场景，LLM仅处理复杂10%
- 🔄 **闭环进化**：学习 → 规则化 → 注入 全自动
- 🛡️ **零丢失通信**：飞书消息100%到达，无静默丢弃

### 2.1 Phase A — 通信可靠性修复 (P0, 预计1天)

> 解决飞书群新成员@机器人无响应问题

#### A1. 修复群消息静默丢弃

**问题**: `bot.ts` 中群组白名单检查失败时直接 `return`，无任何日志或反馈。

**方案**:
```typescript
// 当前行为: 静默丢弃
if (!groupAllowed) {
  log(`feishu[${account.accountId}]: sender ${ctx.senderOpenId} not in group allowlist`);
  return;  // 新成员在非白名单群@bot → 无响应
}

// V6修复: 增加友好提示
if (!groupAllowed) {
  log(`feishu[${account.accountId}]: group ${ctx.chatId} not in allowlist, sending hint`);
  await sendMessageFeishu({
    cfg, to: feishuTo,
    text: "⚠️ 当前群组未启用机器人服务，请联系管理员添加。",
    replyToMessageId: ctx.messageId,
    accountId: account.accountId,
  });
  return;
}
```

#### A2. 增加 groupAllowFrom 通配支持

**问题**: 当前 `groupAllowFrom` 只能精确匹配群ID，每新建一个群都要手动添加。

**方案**: 在 `openclaw.json` 中支持 `"groupPolicy": "open"` 或在 `groupAllowFrom` 中添加 `"*"` 通配符（`policy.ts` 已支持 `"*"` 通配）。

```json
{
  "channels": {
    "feishu": {
      "groupPolicy": "open",  // 或保留allowlist但加入更多群ID
      "requireMention": true   // 群内必须@才响应，保证安全
    }
  }
}
```

#### A3. Lane 超时降级优化

**问题**: `LANE_DISPATCH_TIMEOUT_MS = 90_000` 过长，前面的消息超时会阻塞后续消息。

**方案**:
- 将 `LANE_DISPATCH_TIMEOUT_MS` 从 90s 降到 45s
- 增加 lane overflow 检测：当 queue depth > 3 时，向用户发送"系统繁忙"提示
- 新增 `priority lane`：管理员消息优先调度

#### A4. DM pairing 体验优化

**问题**: 新成员私信机器人触发 pairing 流程，用户不理解配对码。

**方案**: 修改 pairing 提示文案为中文，增加操作指引：
```
"你好！我是 OpenClaw 机器人。首次私信需要管理员授权。
请将以下配对码发送给管理员：{code}
管理员确认后即可正常使用。
也可以在群里直接 @我 提问。"
```

### 2.2 Phase B — 认知内核重构 (P0, 预计3天)

> 将 V5 的串行管道重构为统一认知内核，大幅降低响应延迟

#### B1. 新建 `cognitive-kernel-v6.py` — 替代 v5-orchestrator

**核心改变**:

| | V5 (v5-orchestrator) | V6 (cognitive-kernel) |
|---|---|---|
| 架构 | 串行管道: analyze→plan→gate→finalize | 统一内核: 单次调用完成 |
| 规则引擎 | 5条快速规则 | 50+条规则(自动学习扩展) |
| LLM调用 | 每阶段独立调LLM (4-6次) | 单次LLM调用(带structured output) |
| 上下文 | 无历史感知 | 携带最近5条对话历史 |
| 延迟 | P50=6s | 目标P50<1.5s (规则命中<50ms) |

**实现要点**:

```python
class CognitiveKernel:
    def __init__(self):
        self.rule_engine = RuleEngine()      # 50+条规则，覆盖90%场景
        self.memory_index = MemoryIndex()     # 语义记忆检索
        self.conversation_buffer = {}         # 对话历史缓存

    def process(self, user_input: str, session_id: str) -> dict:
        # Phase 1: 规则引擎快速路径 (<50ms)
        rule_result = self.rule_engine.match(user_input)
        if rule_result and rule_result.confidence > 0.8:
            return rule_result

        # Phase 2: 上下文增强 (<100ms)
        history = self.conversation_buffer.get(session_id, [])[-5:]
        memory_hits = self.memory_index.search(user_input, top_k=3)

        # Phase 3: 单次LLM调用，structured output
        return self.llm_analyze(user_input, history, memory_hits)
```

#### B2. 规则引擎自动扩展 — `rule-engine-v6.py`

**核心改变**: 从手动维护5条规则 → 自动学习+人工审核

- 从 `error-tracker.json` 的 44 条错误中提取防护规则

class ParallelReasoningEngine:
    def reason(self, topic: str, template: str = "auto") -> dict:

        template = self.auto_select_template(topic)

### 2.6 Phase F — 基础设施加固 (P2, 预计1天)

## 第四部分：飞书无响应问题 — 详细排查建议

### 立即排查步骤

```bash
# 1. 查看飞书插件日志，搜索被丢弃的消息
openclaw gateway logs | grep "not in group allowlist"
openclaw gateway logs | grep "not in.*allowlist"
openclaw gateway logs | grep "blocked unauthorized"

# 2. 检查当前群白名单配置
cat ~/.openclaw/openclaw.json | python3 -c "import sys,json; c=json.load(sys.stdin); print(json.dumps(c['channels']['feishu'], indent=2))"

# 3. 检查 dispatch lane 积压
openclaw gateway logs | grep "lane acquired"
openclaw gateway logs | grep "Lane dispatch timeout"

# 4. 检查权限错误
openclaw gateway logs | grep "permission error"
openclaw gateway logs | grep "99991672"
```

### 快速修复 (无需代码改动)

**如果确认是群白名单问题**:
```json
// openclaw.json — 将 groupPolicy 改为 open + requireMention
{
  "channels": {
    "feishu": {
      "groupPolicy": "open",
      "requireMention": true
    }
  }
}
```

**如果确认是 DM pairing 问题**:
```json
// 添加新成员的 open_id 到 allowFrom
{
  "channels": {
    "feishu": {
      "dmPolicy": "open",
      "allowFrom": ["ou_3665e9a9d79f52105fd5868bae4a8b49"]
    }
  }
}
```
